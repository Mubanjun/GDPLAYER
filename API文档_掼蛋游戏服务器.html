1. 总览与认证机制

掼蛋游戏服务器提供 4 个核心 HTTP 接口，覆盖完整的对局生命周期：

接口方法用途调用时机
/join_gamePOST加入对局，获取 game_id每局开始
/check_game/{id}/GET轮询对局状态每 1-5 秒
/play_game/{id}/GET提交出牌我方回合
/rankGET (HTML)查看排行榜按需
注意：同一用户的 join_game 总是返回同一个活跃 game_id（持久会话），而非创建新对局。服务端维护用户会话，多次调用 join 等同于重连。

1.1 RSA 密码加密

密码不可明文传输，必须先经 RSA 公钥加密。客户端使用以下参数：

参数值说明
rsa_e65537公钥指数
rsa_n13526182891679194670531...（2048位）模数（见 config.cfg）

加密算法

# Python 实现（源自 engine.py）
random_part = random_string(8)              # 8位随机字母+数字
plaintext = f"{password}\0{random_part}"    # 密码 + \0 + 随机串
plaintext_bytes = plaintext.encode("utf-8")
password_int = int.from_bytes(plaintext_bytes, "big")
if password_int >= n:
    password_int = password_int % (n - 1)   # 防溢出
encrypted_int = pow(password_int, e, n)     # RSA 加密
rsa_pass = format(encrypted_int, "x")       # 输出十六进制字符串
关键点：每次加密使用不同的随机串，因此同一密码每次加密结果不同。服务端容忍这种随机性（随机串作为 padding 处理）。

1.2 鉴权参数

所有需要鉴权的接口均通过 URL Query 参数传递身份：

?user={user}&password={rsa_pass}
参数类型说明
userstring用户名（学号格式，如 0241123590）
passwordstringRSA 加密后的十六进制字符串

2. 加入对局 — POST /join_game

POST /join_game?user={user}&password={rsa_pass}&humans={0|1}

加入或重连对局，返回当前活跃的 game_id。同一用户多次调用返回同一会话。

请求参数

参数类型必填说明
userstring必填用户名
passwordstring必填RSA 加密密码
humans0 | 1可选是否人类模式（1=人类手动出牌，0=AI 自动）。默认 0

请求示例

POST http://183.175.14.145:8004/join_game?user=0241123590&password=a1b2c3...&humans=0

成功响应

{
  "game_id": 18142
}
字段类型说明
game_idint对局唯一标识，后续接口均需此 ID

失败情况

场景表现
密码错误HTTP 非 200 或返回错误 JSON
服务器无响应连接超时（建议重试 6 次，每次间隔 2s）
返回无 game_idJSON 中无 game_id 字段 → 会话异常
调用建议：join 后服务端会重置 check_game 的计时器。若正在对局中，重复 join 不会中断当前对局，但可能影响服务端超时计算。

3. 查询对局状态 — GET /check_game

GET /check_game/{game_id}/?user={user}&password={rsa_pass}

轮询对局完整状态。这是最核心的接口，包含手牌、回合、桌面牌等全部信息。建议每 1-5 秒调用一次。

路径参数

参数类型说明
game_idintjoin_game 返回的对局 ID

Query 参数

参数类型必填说明
userstring必填用户名
passwordstring必填RSA 加密密码

请求示例

GET http://183.175.14.145:8004/check_game/18142/?user=0241123590&password=a1b2c3...

3.1 返回字段详解

{
  "completed": false,
  "started": true,
  "level": "9",
  "your_seat": 0,
  "your_team": 0,
  "is_your_turn": true,
  "your_hand": ["S3","D3","H5","S5","D7","C9","HK","SK","CA","DA","R","B"],
  "last_play": ["S8"],
  "last_player": 2,
  "current_turn": 0,
  "hand_counts": [12, 8, 10, 12],
  "seats": ["玩家1","玩家2","玩家3","玩家4"],
  "teams": [0, 1, 0, 1],
  "ranking": [],
  "winner_team": -1,
  "trick_history": [[3,["H4"]],[0,["S6"]],[1,["H8"]],[2,["SA"]],[3,[]]]
}
字段类型说明
completedbool对局是否已结束
startedbool对局是否已开始（4人就绪）。信任此字段判断开局
levelstring当前级牌（"2"~"A"）。红桃级牌为逢人配（Wild）
your_seatint (0-3)你的座位号
your_teamint (0-1)你的队伍号
is_your_turnbool是否轮到你出牌
your_handstring[]你的手牌列表（牌面编码见第7节）
last_playstring[]桌面最后有效出牌。空数组=自由领出
last_playerint (-1~3)最后出牌者的座位号。-1=无人出过牌
current_turnint (-1~3)当前轮到的座位号
hand_countsint[4]4位玩家剩余手牌数。初始 27 张
seatsstring[4]4位玩家显示名
teamslist | dict队伍信息（见 3.2）
rankingstring[]名次列表（对局结束后填充，如 ["玩家4","玩家1","玩家3","玩家2"]）
winner_teamint (-1|0|1)获胜队伍。-1=未结束
trick_historyarray本局出牌历史（见 3.3）
0 值陷阱：your_seat=0、your_team=0、last_player=0 均是合法值。解析时必须用 is not None 判断，不可用 or -1（Python 中 0 为 falsy）。

3.2 teams 字段格式

teams 可能返回两种格式，客户端需兼容处理：

格式一：列表（常见）

"teams": [0, 1, 0, 1]   // 座位 0,2 为队 0；座位 1,3 为队 1
若返回 [0, 0, 0, 0] 表示服务端未设置，应按标准掼蛋座位推算：[0, 1, 0, 1]（对角为队友）。

格式二：字典

"teams": {
  "team0": ["玩家1", "玩家3"],
  "team1": ["玩家2", "玩家4"]
}

需结合 seats 字段将成员名映射回座位索引。

3.3 trick_history 结构

出牌历史为二维数组，每个元素 [seat, cards]：

"trick_history": [
  [3, ["H4"]],     // 座位3 出了 红桃4
  [0, ["S6"]],     // 座位0 出了 黑桃6
  [1, ["H8"]],     // 座位1 出了 红桃8
  [2, ["SA"]],     // 座位2 出了 黑桃A
  [3, []],         // 座位3 pass
  [0, []]          // 座位0 pass
]

cards 为空数组表示 pass。可通过遍历 trick_history 的逆序推导 last_play（当 last_play 字段不准确时）。

4. 出牌 — GET /play_game

GET /play_game/{game_id}/?user={user}&password={rsa_pass}&coord={json}

提交出牌决策。注意此接口使用 GET 方法，出牌内容通过 query 参数传递。

路径参数

参数类型说明
game_idint对局 ID

Query 参数

参数类型必填说明
userstring必填用户名
passwordstring必填RSA 加密密码
coordstring (JSON)必填出牌内容，JSON 编码的牌面数组

4.1 coord 出牌格式

coord 是 JSON 编码的字符串，内容为牌面数组：

场景coord 值说明
出单张["S8"]出黑桃8
出对子["S8","H8"]出一对8
出三带二["S8","H8","D8","C4","S4"]三张8带一对4
出炸弹["S8","H8","D8","C8"]四张8炸弹
出大王炸["B","B","R","R"]四大天王（最强炸弹）
pass[]空数组=过牌

完整请求示例

GET http://183.175.14.145:8004/play_game/18142/?user=0241123590&password=a1b2c3...&coord=%5B%22S8%22%5D

# coord 原始值: ["S8"]
# URL 编码后:   %5B%22S8%22%5D
领出时不可 pass：当 last_play 为空（自由领出）时，coord=[] 会被服务端拒绝。必须出至少一张牌。

成功响应

{
  "is_success": true
}

失败响应

{
  "is_success": false,
  "error": "cannot beat"        // 或 "message": "..."
}

4.2 错误分类与处理策略

服务端返回的 error 字段为自然语言描述，客户端需匹配关键字进行分类处理。以下是经源码验证的错误分类表：

error 关键字含义建议处理
cannot pass when leading领出时不能 pass重新选牌出
cannot beat无法压过桌面牌换一组能压的牌，或 pass
invalid无效出牌（牌型不合法）重新生成可行牌
already played牌已出过重新同步手牌后重试
duplicate重复出牌重新选牌
wrong turn轮次错误重新 check_game 同步状态
not your turn不是你的回合重新 check_game 同步状态
game over对局已结束终止本局
already finished对局已结束终止本局
not found游戏不存在终止本局
too many出牌数量不对重新选牌
card not in hand牌不在手牌中重新 check_game 同步手牌
死循环防护：连续被拒 ≥6 次应强制重启对局。重试时务必重新 check_game 获取最新手牌和桌面状态，避免使用过期的本地缓存。

6. 对局生命周期

POST /join_game → 获取 game_id → GET /check_game ↻ 轮询等待 started=true
started=true → GET /check_game → is_your_turn?
YES → GET /play_game → is_success? → 回到轮询
NO → 等待 1-5s → 回到轮询
completed=true → 对局结束 → winner_team == your_team ? → 胜/负

典型时序

客户端                          服务器
  │                               │
  │── POST /join_game ───────────→│  加入对局
  │←── {"game_id": 18142} ───────│
  │                               │
  │── GET /check_game/18142/ ───→│  轮询（未开始）
  │←── {"started": false, ...} ──│
  │                               │
  │   (等待 2-5 秒)                │
  │                               │
  │── GET /check_game/18142/ ───→│  轮询（已开始）
  │←── {"started": true,          │
  │      "is_your_turn": true,    │
  │      "your_hand": [...], ...} │
  │                               │
  │── GET /play_game/18142/      │  出牌
  │    ?coord=["S8"] ───────────→│
  │←── {"is_success": true} ─────│
  │                               │
  │   (等待 1-2 秒，继续轮询)      │
  │                               │
  │   ... 循环出牌 ...             │
  │                               │
  │── GET /check_game/18142/ ───→│  轮询（结束）
  │←── {"completed": true,        │
  │      "winner_team": 0,        │
  │      "ranking": [...], ...} ──│
  │                               │

7. 牌面编码规范

普通牌

格式：{花色}{点数}，共 52 张。

花色编码说明
黑桃SSpade
红桃HHeart（红桃级牌为逢人配）
方块DDiamond
梅花CClub
点数编码排序值
2 ~ 92-90-7
10T8
杰克J9
皇后Q10
国王K11
王牌A12

特殊牌

编码名称说明
B小王Black Joker
R大王Red Joker（最大单牌）

逢人配（Wild Card）

定义：红桃花色的当前级牌。例如级牌为 "9" 时，H9 即为逢人配，可替代任意牌组成合法牌型。

排序

牌面排序优先级（从小到大）：

普通牌(2~A) < 级牌(4张) < 小王(B) < 大王(R)
同点数: S < H < D < C

牌型类型

类型英文名说明
单张single1 张
对子pair2 张同点数
三张triple3 张同点数
三带二full_house3+2（三张+对子）
顺子straight5+ 连续单张
连对plate3+ 连续对子
连三steel2+ 连续三张
炸弹bomb4+ 张同点数
同花顺straight_flush5+ 连续同花色
四大天王quad_kingsB,B,R,R（最强炸弹）

8. 错误处理参考表

场景HTTP 状态处理策略
密码错误 / 鉴权失败非 200终止，检查 config.cfg 中 rsa_e/rsa_n
game_id 不存在200 + error: "not found"重新 join_game
网络超时无响应重试（join 6次, check 5次, play 2次）
对局未开始200 + started: false等待 2-5s 后重新 check
非我方回合200 + is_your_turn: false等待 1-5s 后重新 check
出牌被拒200 + is_success: false按 error 关键字分类处理（见 4.2）
对局已完成200 + completed: true读取 winner_team 判断胜负
卡在开局阶段200 + all hand_counts=27等待 >5min 视为卡死，退出
超时建议：单局超时设为 5-30 分钟。check_game 的 HTTP 超时设为 30s，play_game 设为 30-120s（服务端出牌确认有时较慢）。
